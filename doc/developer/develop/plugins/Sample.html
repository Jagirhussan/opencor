The purpose of this plugin is to give other plugins access to a simple add function, which could be used as follows: 

#include "sampleutils.h"

...

double addResult = Sample::add(3.0, 5.0);

...

In order to create such a plugin, we first need to decide on its category. Currently supported categories are Analysis, Api, Editing, Miscellaneous, Organisation, Sample, Simulation, Support, ThirdParty and Widget (see the Category enum in the PluginInfo class in [OpenCOR]/src/plugins/plugininfo.h). For our current plugin, we will be using the Sample category.

The category of a plugin is not only needed by OpenCOR (to improve user experience), but also to determine the location of a plugin's code. Indeed, the code of all our plugins is located under [OpenCOR]/src/plugins/ and, more precisely, under a category folder. Thus, the code of our Sample plugins is to be located under the sample folder (see [OpenCOR]/src/plugins/sample/). However, we need to be able to distinguish between the code of our various Sample plugins. We therefore need to create a folder for our plugin and its name is to be that of our plugin, i.e. Sample. So, in the end, the code of our Sample plugin is to be located under [OpenCOR]/src/plugins/sample/Sample/.

Now, OpenCOR is built using CMake, so to be able to build our plugin, we need to have a simple CMake project for it. This is done by creating a CMakeLists.txt file (see [OpenCOR]/src/plugins/sample/Sample/CMakeLists.txt), which contents is as follows: 

PROJECT(SamplePlugin)

# Add the plugin

ADD_PLUGIN(Sample
    SOURCES
        ../../plugininfo.cpp

        src/sampleplugin.cpp
        src/sampleutils.cpp
    INCLUDE_DIRS
        src
    QT_MODULES
        Core
    QT_LIBRARIES
        QtCore
)

The first line specifies the name of the CMake project for our Sample plugin, which here is SamplePlugin. Next, we have a call to the ADD_PLUGIN macro. This macro effectively handles the building of our pugin, this using the information that is passed to it. That information comes in the form of a series of parameters, some of which are keywords (SOURCES, INCLUDE_DIRS, QT_MODULES and QT_LIBRARIES at lines 6, 11, 13 and 15) that are used to give a specific meaning to the parameters that follow them. (The meaning of those keywords and that of others can be found here).

As for any plugin, our Sample plugin must reference plugininfo.cpp (line 7), so that it can provide some basic information about itself (more on this below). The .cpp files that contain the plugin's code clearly need to be referenced too (lines 9 and 10). Note that they start with sample, i.e. the name of the plugin. This is a convention that is used throughout OpenCOR's code to ensure that there is no name clashing. To make #include statements simpler to write (and to maintain), we provide src as an include directory (line 12). Finally, and though our Sample plugin is very minimal, OpenCOR uses the Qt framework. So, we need to reference both the Core module and the QtCore library (lines 14 and 16, respectively).
