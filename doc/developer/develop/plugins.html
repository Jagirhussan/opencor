OpenCOR relies on a plugin approach. Some of its plugins are used to provide access to third-party libraries or APIs while others are used to organise, edit or simulate files, or are plugins that do not fit anywhere else and are therefore available under the miscellaneous category.

Namespace

A plugin's code must be in its own namespace within the OpenCOR namespace. For example, for the Core plugin (see [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp and [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h), we have: 

...
namespace OpenCOR {
namespace Core {
...
}   // namespace Core
}   // namespace OpenCOR
...

Class definition

We rely on Qt's support for plugins, which means that, for each plugin, we must have at least three files: a .cpp, a .h and a .json file (e.g. [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp, [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h and [OpenCOR]/src/plugins/misc/Core/src/coreplugin.json).

The JSON file simply contains a key to the plugin, which in the case of the Core plugin looks as follows: 

{
    "Keys": [ "CorePlugin" ]
}

Each plugin must inherit from QObject, refer to its JSON file, and may implement any of the core, file, GUI, internationalisation and solver interfaces.

For example, the Core plugin implements the core, GUI and internationalisation interfaces, resulting in the following definition (see [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h): 

...
class CorePlugin : public QObject, public CoreInterface, public GuiInterface,
                   public I18nInterface
{
    Q_OBJECT

    Q_PLUGIN_METADATA(IID "OpenCOR.CorePlugin" FILE "coreplugin.json")

    Q_INTERFACES(OpenCOR::CoreInterface)
    Q_INTERFACES(OpenCOR::GuiInterface)
    Q_INTERFACES(OpenCOR::I18nInterface)

public:
#include "coreinterface.inl"
#include "guiinterface.inl"
#include "i18ninterface.inl"
...
};
...

On the other hand, the LLVM plugin does not need to implement any interface since its sole purpose is to provide access to LLVM (as well as Clang; see [OpenCOR]/src/plugins/3rdparty/LLVM/src/llvmplugin.h), hence its simpler definition: 

...
class LLVMPlugin : public QObject
{
    Q_OBJECT

    Q_PLUGIN_METADATA(IID "OpenCOR.LLVMPlugin" FILE "llvmplugin.json")
};
...

Basic information

For a plugin to be recognisable by OpenCOR, it must provide the following information (see [OpenCOR]/src/plugins/plugininfo.h):

  • Category: plugins are grouped in different categories. These are only meant to improve user experience and therefore have no effects on the plugin itself.
  • CLI support: specify whether the plugin offers command line interface (CLI) support.
  • Manageable: specify whether a user can change the loading state of a plugin. As a rule of thumbs, if a plugin can be used by another plugin, then it probably should not be made manageable.
  • Dependencies: some plugins may depend on other plugins. OpenCOR must therefore know about them, so that it can make sure that they get loaded before the plugin itself.
  • Descriptions: a short description of the plugin in various languages.

This information is made available to OpenCOR through a function, which in the case of the Core plugin is declared as follows (see [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h): 

PLUGININFO_FUNC CorePluginInfo();

As you can see, the name of the function is related to the name of the plugin and is <PluginName>PluginInfo.

Note: to ensure the uniqueness of a plugin, OpenCOR uses its filename to determine the name of its function. So, make sure that they match or OpenCOR will not be able to recognise the plugin.

The body of such a function is straightforward and looks as follows in the case of the Core plugin (see [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp): 

PLUGININFO_FUNC CorePluginInfo()
{
    Descriptions descriptions;

    descriptions.insert("en", QString::fromUtf8("the core plugin."));
    descriptions.insert("fr", QString::fromUtf8("l'extension de base."));

    return new PluginInfo(PluginInfo::Miscellaneous,
                          false,
                          false,
                          QStringList(),
                          descriptions);
}

Support for the internationalisation of a plugin's description would normally be done using Qt's tr() function, but the C nature of the function means that it cannot be done. So, instead, we use a QMap-based approach.

Import/export

To support plugin dependencies on Windows requires some classes and/or functions to be imported/exported (using Q_DECL_IMPORT/Q_DECL_EXPORT, respectively). To handle this, we must create a header file that defines a macro that refers to either Q_DECL_IMPORT or Q_DECL_EXPORT, depending on how the plugin code is to be compiled. The Core plugin is a bit special in that respect (see [OpenCOR]/src/plugins/misc/Core/src/coreglobal.h), but in the case of the Compiler plugin it looks as follows (see [OpenCOR]/src/plugins/misc/Compiler/src/compilerglobal.h): 

...
#ifdef _WIN32
    #ifdef Compiler_PLUGIN
        #define COMPILER_EXPORT __declspec(dllexport)
    #else
        #define COMPILER_EXPORT __declspec(dllimport)
    #endif
#else
    #define COMPILER_EXPORT
#endif
...

_WIN32 and Compiler_PLUGIN (or, more generally, <PluginName>_PLUGIN) are automatically defined at build time and are used to determine the value of COMPILER_EXPORT (or, more generally, the value of <PLUGINNAME>_EXPORT). COMPILER_EXPORT can then be used as follows (see [OpenCOR]/src/plugins/misc/Compiler/src/compilerengine.h): 

...
class COMPILER_EXPORT CompilerEngine : public QObject
{
...
};
...

ADD_PLUGIN macro

The CMake ADD_PLUGIN macro is used both to build and to package a plugin (see [OpenCOR]/cmake/common.cmake) and it requires one or several types of arguments to be passed to it:

  • SOURCES: impementation files.
  • HEADERS_MOC: header files, which define at least one QObject-based class.
  • UIS: user interface files.
  • INCLUDE_DIRS: various locations where header files can be found.
  • DEFINITIONS: definitions needed to build the plugin (e.g. [OpenCOR]/src/plugins/3rdparty/SUNDIALS/CMakeLists.txt).
  • PLUGIN_DEPENDENCIES: other plugins needed by the plugin (e.g. [OpenCOR]/src/plugins/misc/Help/CMakeLists.txt).
  • PLUGIN_BINARY_DEPENDENCIES: the binary version of other plugins needed by the plugin (e.g. [OpenCOR]/src/plugins/misc/Compiler/CMakeLists.txt).
  • QT_MODULES: Qt modules needed by the plugin.
  • QT_DEPENDENCIES: Qt libraries needed by the plugin.
  • EXTERNAL_BINARY_DEPENDENCIES_DIR: location of any external binaries needed by the plugin (e.g. [OpenCOR]/src/plugins/api/CellMLAPI/CMakeLists.txt).
  • EXTERNAL_BINARY_DEPENDENCIES: external binaries needed by the plugin (e.g. [OpenCOR]/src/plugins/api/CellMLAPI/CMakeLists.txt).
  • EXTERNAL_LIBRARY_DEPENDENCIES: external libraries needed by the plugin (e.g. [OpenCOR]/src/plugins/3rdparty/LLVM/CMakeLists.txt.build).
  • TESTS: tests for the plugin (e.g. [OpenCOR]/src/plugins/misc/Compiler/CMakeLists.txt).

Note: due to the intrinsic nature of OS X binaries, it is critical that you list all the Qt libraries needed by a plugin. Failure to do so may result in OpenCOR not behaving as expected.
